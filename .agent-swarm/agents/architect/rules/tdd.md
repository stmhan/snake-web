# TDD 워크플로우

## Red-Green-Refactor 사이클

모든 프로덕션 코드는 실패하는 테스트에서 시작한다.

1. **RED** — 실패하는 테스트를 작성한다. 테스트가 올바른 이유로 실패하는지 확인한다.
2. **GREEN** — 테스트를 통과시키는 가장 단순한 코드를 작성한다. 과도한 설계를 하지 않는다.
3. **REFACTOR** — 테스트가 통과하는 상태에서 코드를 정리한다. 중복 제거, 명명 개선, 구조 정리.

이 사이클을 기능 하나가 완성될 때까지 반복한다.

## 테스트 작성 순서

1. 가장 단순한 정상 경로(happy path)부터 시작
2. 엣지 케이스 추가 (빈 입력, 경계값, null)
3. 에러 경로 추가 (잘못된 입력, 네트워크 실패)
4. 필요하면 통합 테스트 추가

## 규칙

- 실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다.
- 한 번에 하나의 테스트만 실패하게 한다. 여러 테스트를 한꺼번에 작성하지 않는다.
- 리팩토링 단계에서는 새로운 기능을 추가하지 않는다.
- 테스트가 깨지면 프로덕션 코드 추가를 멈추고 테스트부터 고친다.
