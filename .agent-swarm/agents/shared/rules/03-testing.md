# 테스트

## 원칙

- **동작을 테스트한다.** 내부 구현이 바뀌어도 동작이 같으면 테스트는 통과해야 한다.
- **한 테스트에 하나의 검증.** 테스트가 실패했을 때 무엇이 깨졌는지 바로 알 수 있어야 한다.
- **테스트 이름은 기대 동작을 서술한다.** `test_login` (X) → `test_잘못된_비밀번호로_로그인하면_401을_반환한다` (O)

## 구조

Arrange-Act-Assert 패턴을 따른다:

```
# Arrange: 테스트에 필요한 상태를 설정
# Act: 테스트 대상 동작을 실행
# Assert: 결과를 검증
```

테스트 파일은 구현 파일과 같은 디렉토리에 둔다. 별도 `__tests__/` 디렉토리를 만들지 않는다.

## 모킹

- 외부 의존성(네트워크, DB, 파일시스템)은 모킹한다.
- 내부 모듈은 모킹하지 않는다. 내부 모듈을 모킹해야 한다면 설계를 재고한다.
- 모킹은 테스트 대상의 경계에서만 적용한다.

## 커버리지

- 새 코드에는 반드시 테스트를 작성한다.
- 버그 수정에는 해당 버그를 재현하는 회귀 테스트를 포함한다.
- 100% 커버리지를 목표하지 않는다. 핵심 비즈니스 로직과 엣지 케이스를 우선한다.

## 금지 사항

- 테스트 간 상태 공유: 각 테스트는 독립적으로 실행 가능해야 한다.
- sleep/delay 기반 테스트: 비결정적 테스트(flaky test)의 원인이 된다.
- 네트워크 호출: 단위 테스트에서 실제 네트워크 요청을 보내지 않는다.
