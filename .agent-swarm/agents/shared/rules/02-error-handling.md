# 에러 처리

## 원칙

외부 입력(사용자 입력, API 응답, 파일 I/O)은 항상 검증한다. 내부 함수 호출에서 방어적 코딩은 불필요하다.

에러는 발생 지점에서 가장 가까운 곳에서 처리한다. 상위 호출자에게 넘기는 것이 더 적절하면 명시적으로 전파한다.

## 해야 할 것

- 비동기 작업에는 try/catch를 사용하되, catch 블록에서 구체적인 에러 유형을 구분한다.
- 에러 메시지에 실패한 작업과 입력값을 포함한다: `"사용자 ID 123 조회 실패"` (O) vs `"에러 발생"` (X)
- 리소스(파일 핸들, DB 연결 등)는 finally에서 정리한다.
- 복구 가능한 에러와 불가능한 에러를 구분한다. 복구 불가능하면 빠르게 실패한다.

## 하지 말아야 할 것

- 빈 catch 블록으로 에러를 삼키지 않는다. 최소한 로깅은 한다.
- 제어 흐름에 예외를 사용하지 않는다. 예외는 예외적 상황에만 쓴다.
- 최종 사용자에게 스택 트레이스나 내부 에러를 노출하지 않는다.
- 변경 없이 catch-rethrow하지 않는다. 컨텍스트를 추가하거나 catch를 제거한다.
