# 코드 리뷰

## 리뷰 출력 형식

응답의 **첫 줄**은 반드시 다음 중 하나:
- `LGTM` — 코드가 머지 가능한 상태
- `CHANGES REQUESTED` — 수정이 필요함

리뷰 본문은 **한국어**로 작성한다.

## 리뷰 기준

### 보안 (반드시 확인)

- 하드코딩된 비밀키, API 키, 토큰, 비밀번호가 없는가
- SQL 인젝션, XSS, 커맨드 인젝션 가능성이 없는가
- 사용자 입력을 검증 없이 사용하는 곳이 없는가
- 인증/인가 검사가 누락된 엔드포인트가 없는가

### 정확성

- 로직이 요구사항과 일치하는가
- 엣지 케이스를 처리하는가 (빈 배열, null, 경계값)
- 동시성 문제가 없는가 (레이스 컨디션, 데드락)
- 리소스 누수가 없는가 (파일 핸들, DB 연결, 이벤트 리스너)

### 테스트

- 새 코드에 대한 테스트가 존재하는가
- 테스트가 동작을 검증하는가 (구현 세부사항이 아닌)
- 엣지 케이스와 에러 경로를 커버하는가
- 테스트가 결정적인가 (매번 같은 결과)

### 설계

- 함수와 클래스가 단일 책임 원칙을 따르는가
- 추출해야 할 중복 로직이 없는가
- 기존 프로젝트 패턴과 일관성을 유지하는가
- 불필요한 복잡성을 추가하지 않았는가

### 가독성

- 변수명과 함수명이 의도를 드러내는가
- 복잡한 로직에 "왜"를 설명하는 주석이 있는가
- 미사용 코드, 미사용 import, 디버그 로그가 없는가

## 피드백 작성 요령

- 파일 경로와 줄 번호를 명시한다: `src/api/user.py:42`
- 문제점과 함께 해결 방안을 제시한다
- 심각도를 구분한다: `[필수]` 반드시 수정, `[제안]` 개선 권장, `[질문]` 확인 필요
- 잘 작성된 코드에 대해서도 구체적으로 언급한다
